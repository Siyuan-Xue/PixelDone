# Project Engineering Specification

This document governs the engineering workflow, MVP development, iteration, verification, and release decisions for the PixelDone Android repository.

This document does not define product-line positioning, product naming, developer identity, or visual style. Those belong to `PRODUCT_LINE_SPEC.md` and `DESIGN_SPEC.md`.

## 1. Engineering Philosophy

Every project starts from a real idea, not from technical showmanship. The implementation must serve the fastest possible validation of that idea.

Core principles:

1. Prove the tool is useful before expanding its capabilities.
2. Complete one core path before adding branches.
3. Keep the implementation clear before pursuing abstraction.
4. Target modern Android devices by default; do not sacrifice implementation clarity for old-device coverage.
5. Review every scaling step for necessity, elegance, and maintainability.

## 2. Project Origin

Every new project starts from a brainstorm or an already-formed useful idea. Before coding, record these three sentences:

- Core idea: what specific problem this tool solves.
- Target users: who will use it on a phone or tablet.
- MVP feature: the single most necessary usable workflow.

If the MVP cannot be explained in one sentence, the project is not yet focused enough to start.

## 3. Android Studio GUI Project Creation

All new Android projects should be created through the Android Studio GUI by default, unless the project has a clear reason to use another scaffold.

### 3.1 Entry Point

If you are on the welcome screen:

1. Click `New Project` or `Start a new Android Studio project`.

If a project is already open:

1. Choose `File > New > New Project`.

### 3.2 Template

1. Select `Phone and Tablet`.
2. Select `Empty Activity`.
3. Click `Next`.

`Empty Activity` is the default starting point because it provides a modern declarative UI foundation with Jetpack Compose and Material 3. Do not use more complex templates unless maintaining a legacy codebase.

### 3.3 Project Configuration

Default configuration:

- `Name`: follow the product naming rules in `PRODUCT_LINE_SPEC.md`.
- `Package name`: follow the package naming rules in `PRODUCT_LINE_SPEC.md`.
- `Save location`: any local directory chosen by the developer. Do not encode a developer's machine path into the repository.
- `Language`: `Kotlin`.
- `Minimum SDK`: choose the modern baseline available in the current stable Android Studio / Android Gradle Plugin template. Do not lower it just to cover old devices. If a core official capability requires a higher API level, prefer raising `minSdk` over adding compatibility branches.
- `Compile SDK` / `Target SDK`: use the current stable Android SDK and the forward configuration supported by the current AGP toolchain. Follow toolchain updates proactively.
- `Build configuration language`: prefer Kotlin DSL.

After configuration, click `Finish` and wait for Gradle sync.

## 4. Git Check After GUI Creation

After a new project is created through Android Studio GUI and the user tells Codex that the project exists, Codex must not start MVP development immediately. Before editing business code, first inspect the initial Git configuration and state.

Minimum checks:

- Run `git status --short` from the repository root and identify which files were newly generated by Android Studio and which are existing user changes.
- Check whether the new subproject root and the app module have suitable `.gitignore` files.
- Confirm `.gitignore` excludes common Gradle/Android Studio generated files, such as `.gradle/`, `build/`, `local.properties`, `*.iml`, and local-only files under `.idea/`.
- Confirm source code, Gradle configuration, resources, README files, wrapper files, and other necessary project files are not ignored by mistake.
- If Android Studio generated `.idea/`, determine which files are shared project configuration and which are local state. When uncertain, stay conservative and explain the risk to the user.
- Before MVP implementation, briefly report the Git initial-state and `.gitignore` conclusion to the user.

Only after the checks above are complete should MVP implementation begin.

## 5. MVP First

After the project scaffold is ready, the first step is always to build a minimum viable product.

An MVP must include:

- One main screen.
- One core input path.
- One core output or result.
- One verifiable success state.
- Necessary error states.
- A lightweight non-default app icon that is actually packaged in the test APK, so the installed app does not show the Android Studio default icon.
- Installation on a modern emulator or the current development device. Old-device coverage is not promised.

An MVP does not include:

- A full settings system.
- Multiple themes, unless theming is the core product.
- Accounts.
- Cloud sync.
- Large-scale history.
- Complex animations.
- Unnecessary third-party SDKs.

## 6. Recommended Engineering Structure

New projects should start as a single-module Android app by default.

Recommended layers:

- `MainActivity.kt`: app entry point and top-level Compose assembly.
- `ui/theme/`: colors, typography, and theme.
- `ui/`: composable UI components and screens.
- `domain/` or root-package core files: pure Kotlin business logic.
- `test/`: unit tests for core logic.

Early projects may keep the structure lightweight. Add packages and abstractions only when files become too large, responsibilities become mixed, or real reuse exists.

## 7. Future-Ready Android Architecture

Growing Android projects should stay ready for future extension without prebuilding architecture for hypothetical features. Requirements drive architecture: before a major restructuring, state the functional requirements and the non-functional requirements that justify it, such as maintainability, security, reliability, testability, performance, privacy, and evolvability.

### 7.1 Default Layer Model

For a growing app, prefer the Android recommended UI / data / optional domain architecture while keeping the project single-module until there is a concrete reason to split modules.

Default package model:

- `MainActivity.kt`: Android entry only. It owns activity lifecycle setup, system bars, permission launchers, and `setContent`, but it must not directly contain business rules, storage code, network code, schedulers, or large screen implementations.
- `Application` plus `di/`: app-level dependency container. Manual dependency injection is acceptable for small apps; move to Hilt only when dependency graph size, lifecycle scoping, or test setup complexity makes manual wiring costly.
- `ui/<feature>/`: feature route, screen, ViewModel, UI state, UI actions, and one-shot system actions. Compose screens render state and emit actions. They must not read or write storage, files, databases, network clients, sensors, receivers, services, or platform providers directly.
- `ui/<feature>/components/`: reusable visual-only Compose components. Components receive values and callbacks; they do not own feature business decisions.
- `domain/<feature>/`: pure Kotlin models, rules, policies, validators, sorters, and use cases. Add use-case classes only when a rule is complex, reused, or needs independent tests. This package must avoid Android and Compose imports.
- `data/<feature>/`: repositories, local data sources, remote data sources, codecs, mappers, and migration helpers. Repositories are the only entry point to data sources and expose stable app data APIs to ViewModels or domain services.
- Android boundary packages such as `reminder/`, `sync/`, `update/`, `export/`, or `backup/`: isolate platform APIs that have lifecycle, permission, file, network, background-work, or installation behavior.

### 7.2 UI State And Data Flow

Prefer unidirectional data flow for non-trivial screens:

1. UI renders immutable UI state.
2. User input becomes a typed UI action.
3. The ViewModel handles the action, calls domain and repository APIs, and updates state.
4. One-shot work such as opening settings, requesting permission, starting install flows, showing external pickers, or launching system UI is represented as a pending system action and consumed once by the Activity or route.

Use lifecycle-aware state collection in Compose, such as `collectAsStateWithLifecycle`, for Flow-backed state. ViewModels should expose state with `StateFlow` when state changes over time, and use coroutines for asynchronous work. Avoid passing mutable collections or storage objects directly into composables.

### 7.3 Extension Gating

Every future extension starts from a user story and acceptance criteria. Use the story to identify the actor, goal, value, success state, and error states before adding new packages or abstractions.

Add a new abstraction only when it:

- Removes real duplication.
- Separates responsibilities that already change for different reasons.
- Isolates volatile platform, permission, storage, network, or file behavior.
- Supports a confirmed extension with stated acceptance criteria.
- Makes important behavior easier to test.

Do not prebuild empty managers, service layers, modules, interfaces, or factories only because a feature might exist someday. Plan for extension points, but implement only the next real requirement.

### 7.4 Future Extension Taxonomy

Use these bounded contexts when PixelDone-like utilities grow:

- Persistence migration: keep the repository API stable. Add data-source adapters, compatibility readers, and migration tests before moving from `SharedPreferences` JSON to DataStore, Room, files, or another store.
- Cloud, sync, and backup: add `data/sync` and domain conflict-resolution rules. Keep the offline-first local repository as the source of truth unless the requirement explicitly changes that product behavior.
- Notifications, reminders, and background work: keep `AlarmManager`, WorkManager, receivers, services, notification channels, and permission checks behind scheduler or platform-boundary interfaces. Domain code decides timing policy; Android boundary code performs scheduling.
- Attachments, export, and import: isolate file, URI, MIME, media decoding, and path-safety handling in data or platform-boundary packages. UI code may request a picker or show previews, but it must not validate paths, persist grants, or parse external file formats directly.
- Multi-screen navigation and settings: keep one Activity by default. Add typed routes, screen-level ViewModels, and explicit state handoff. Do not use direct cross-screen shared mutable state.
- AI or external services: require privacy, failure-mode, rate-limit, prompt-boundary, data-retention, and offline/error behavior rules before implementation. Never send user data to a service without an explicit product requirement and user-facing consent path.
- Monetization, accounts, and remote config: treat each as a separate bounded context with its own repository, risk review, security review, and rollback plan.

### 7.5 Architecture Quality Rules

Apply software engineering principles pragmatically:

- Prefer high cohesion and low coupling. A file, class, composable, or package should have one clear reason to change.
- Use SRP, OCP, ISP, and DIP as design checks, not ceremony. Keep simple code simple, but stop mixing UI, persistence, scheduling, and business rules in one place.
- Use patterns by intent: Repository for data access, Adapter or Facade for Android and external services, Strategy for interchangeable policies, Observer through Flow or StateFlow, and Factory or DI containers for object creation.
- Make code a maintainable asset: readable names, consistent formatting, clear package ownership, and comments that explain architecture boundaries or non-obvious business rules.
- Every architecture change must include a short risk note covering the relevant items: migration risk, permission risk, lifecycle/background risk, storage compatibility risk, security/privacy risk, and user-experience regression risk.

## 8. Implementation Rules

### 8.1 Kotlin First

Use Kotlin for business logic and UI. Prefer clear data classes, pure functions, and immutable state.

### 8.2 Compose First

Use Jetpack Compose for UI. Keep composables small and direct. Avoid building a complex design system too early. Extract a reusable component only after it is reused at least twice.

### 8.3 Separate Business Logic From UI

Calculation, parsing, conversion, validation, and similar logic must not be buried inside composables. Put it in pure Kotlin functions or engines and cover it with unit tests.

### 8.4 Technology Stack

Use the lightest, most authoritative, and most forward Android-native stack by default:

- Kotlin, Jetpack Compose, Material 3, AndroidX, Android Gradle Plugin, and Kotlin DSL are the default combination.
- Prefer Android platform capabilities, Google / AndroidX / Jetpack official libraries, and official recommended practices.
- Before implementing a core capability from scratch, research whether an official, authoritative, lightweight, focused, and current library or platform API already solves it well. Prefer reuse when it reduces complexity, improves correctness, and fits the MVP without pulling in unnecessary product surface.
- Prefer current stable dependencies. Use alpha / beta dependencies only when they solve a core problem, come from official or clearly authoritative sources, and provide benefits that clearly outweigh maintenance risk.
- Do not add cross-platform frameworks, heavy architectures, compatibility layers, polyfills, or shims for old-device coverage.
- When a modern API significantly reduces complexity, prefer raising `minSdk` or narrowing device scope over keeping multiple implementations.

### 8.5 Dependency Restraint

Before adding a dependency, confirm:

- It solves a core problem.
- It is more reliable than a local implementation.
- It will not significantly increase APK size, permissions, or maintenance cost.
- Its capability will actually be used in the MVP or current iteration.
- It has been compared against official platform APIs and other authoritative lightweight options.

## 9. Scaling Review

When proposing further scaling, every new feature must pass this review:

1. Does it serve the original core idea?
2. Can a user understand it within 10 seconds?
3. Can it be merged into the existing interaction instead of adding an isolated entry point?
4. Does it increase state count, permission needs, or data migration cost?
5. Is there a simpler implementation?

If a feature cannot pass the review, postpone it, cut it down, or redesign it.

## 10. Testing And Verification

Minimum MVP-stage testing:

- Unit tests for the core engine.
- Tests for important edge cases.
- At least one `assembleDebug`.
- At least one manual verification on a modern emulator or the current development device.

Additional checks during iteration:

- Unit tests for new features.
- UI-state screenshots or manual verification notes.
- Tablet wide-screen layout check.
- Error-state and empty-state checks.

Architecture testing rules:

- Domain rules require JVM unit tests.
- Repository and data-source behavior requires persistence, compatibility, and migration tests when storage formats or data ownership change.
- ViewModel and unidirectional-data-flow behavior requires action-to-state tests for important user paths and error paths.
- Android boundary code requires focused tests around policy wrappers where practical. When behavior depends on device permissions, alarms, notification UI, installers, or other system surfaces, record manual QA notes.
- Major refactors must pass `testDebugUnitTest` and `assembleDebug`.
- APK-output-rule changes must also confirm both `app-debug.apk` and `{ProjectName}-{versionName}-debug.apk` in `app/build/outputs/apk/debug/`.
- Documentation-only changes do not require a Gradle build unless they change release, build, or verification commands. They should still pass Markdown/diff sanity checks such as `git diff --check`.

## 11. README Requirements

Every subproject must have a `README.md` that includes:

- One-sentence project description.
- Current features.
- Tech stack.
- Build commands.
- Installation method.
- Current status, such as `MVP prototype`, `Internal alpha`, or `Production candidate`.

The README must not exaggerate project maturity. Early projects should be clearly marked as MVPs.

## 12. Delivery Standard

After each implementation, confirm at least:

- The project builds.
- Core logic has tests.
- UI follows `DESIGN_SPEC.md`.
- Product naming and developer identity follow `PRODUCT_LINE_SPEC.md`.
- The test APK uses the current product's lightweight custom app icon, not the default Android icon.
- No unexplained large dependency was added.
- No compatibility branches, complex fallbacks, or outdated stack choices were added for old-device coverage.
- No complex architecture was prebuilt for hypothetical future features.

## 13. Git Wrap-Up And Release Flow

This section owns PixelDone Git wrap-up, local debug APK preparation, beta distribution policy, formal signed APK releases, tagging, GitHub release publishing, and app-update behavior. If release rules continue to grow beyond this workflow, split them into a dedicated `RELEASE_SPEC.md` and keep this section as the engineering entry point.

### 13.1 Release Ownership

PixelDone is released from this independent Git repository. Do not release PixelDone from a parent workspace checkout. Enter the PixelDone repository root before Git wrap-up, tagging, pushing, or creating releases.

### 13.2 Release Repository, Update Source, And Local Git Remote

The canonical code repository, Git push target, tag target, and release publishing surface is GitHub:

```text
https://github.com/Siyuan-Xue/PixelDone
```

All code pushes, tags, formal releases, and beta RC prereleases are created on GitHub unless the user explicitly requests a different one-off operation.

The primary app-internal release and update source is GitHub Releases:

```text
https://github.com/Siyuan-Xue/PixelDone/releases
```

The synced Gitee mirror is the fallback app-internal release and update source:

```text
https://gitee.com/milesxue/PixelDone/releases
```

The user maintains Gitee synchronization from GitHub. Agents should publish to GitHub, then let the configured Gitee sync expose the same releases and APK assets for fallback checks and fallback APK downloads.

Do not change the local Git `origin` URL as part of release-source or update-channel work unless the user explicitly requests it. The expected local `origin` remains the GitHub repository. Use `git remote -v` only to report the current local remote state.

### 13.3 Release Channels

PixelPark uses three separate release channels:

- **Local debug**: build `{ProjectName}-{versionName}-debug.apk` and deploy it directly to an authorized Android device or emulator with `adb`. This channel is for local validation only and is not the default app-internal update target.
- **Internal beta**: publish a GitHub prerelease only when explicitly needed. Use RC tags such as `v2.8.0-rc.1` and attach the matching debug APK asset, such as `PixelDone-2.8.0-rc.1-debug.apk`. Beta app-internal update checks prefer the GitHub prerelease asset and may fall back to the synced Gitee prerelease asset. Prereleases are lightweight distribution, not access control, because anyone with the link may download the asset.
- **Formal release**: build a signed `{ProjectName}-{versionName}-release.apk`, publish it as a normal GitHub Release, and let formal app-internal update checks prefer the signed GitHub release APK asset with synced Gitee as fallback.

Debug prereleases are allowed only for the beta channel. They are not the formal release path for user updates.

### 13.4 Version Ordering And Update Rules

Version rules:

- Formal release tags use `v{major}.{minor}.{patch}`, for example `v1.2.0`.
- Beta RC prerelease tags use `v{major}.{minor}.{patch}-rc.{number}`, for example `v2.8.0-rc.1`.
- Release title: `{ProjectName} v{major}.{minor}.{patch}` for formal releases, or `{ProjectName} v{major}.{minor}.{patch}-rc.{number}` for beta RC prereleases.
- In `0.x.y` versions, increment `x` only for iterative new features and `y` only for bug fixes. Version numbers must be released in semantic order.
- Users do not need forced sequential upgrades. A formal APK can update older formal versions when `applicationId` and signing certificate match and `versionCode` increases.
- Every app module release or deploy build must increase `versionCode` when it needs to update an already installed build through Android package management.
- Beta and debug `versionCode` values must leave a clean path to the next formal release. Do not publish a beta or debug build whose `versionCode` would block a later formal signed update.
- The release build type owns the `formal` app-internal update channel. The debug build type owns the `beta` app-internal update channel. Agents must not manually edit this channel mapping before each build.
- Formal update checks accept only normal releases with `vX.Y.Z` tags and `PixelDone-X.Y.Z-release.apk` assets. Beta update checks accept only prereleases with `vX.Y.Z-rc.N` tags and `PixelDone-X.Y.Z-rc.N-debug.apk` assets. Checks prefer GitHub Releases and fall back to the synced Gitee mirror when GitHub fails or lacks a matching APK asset.
- Before advancing `versionName` or `versionCode` for an app iteration, commit the source state of the previous numbered version if that version was built, installed, distributed, tagged, documented, or otherwise referenced. Every numbered debug, beta, or formal version that leaves the working tree must be reproducible from a reachable commit.
- Advance `versionName` only for a clear semantic unit, such as a completed feature, a distinct bug fix, a beta milestone, or a formal release candidate. Repeated refinements of the same bug fix, or different presentations of the same feature gate, must keep the same `versionName`.
- If Android package-management testing truly requires a new `versionCode` during the same semantic fix cycle, first commit the current numbered state or use a same-version reinstall workflow when acceptable. Never create skipped public versions solely because intermediate numbered debug states were left uncommitted.
- If the target tag/release already exists, first confirm it points to the correct commit. Do not overwrite existing tags, releases, or release assets. If necessary, increment to the next semantic version.

### 13.5 Local Debug APK Rule

Debug APK rules:

- File name is always `{ProjectName}-{versionName}-debug.apk`, for example `PixelCalc-0.2.1-debug.apk`, `PixelDone-1.2.0-debug.apk`, or `PixelRead-0.2.1-debug.apk`.
- `{ProjectName}` must be the official product name defined in `PRODUCT_LINE_SPEC.md`, using the `Pixel<Name>` / `Pixel[xxx]` form. Square brackets are placeholder notation only and must not appear in real file names.
- `{versionName}` must come directly from the current app module's `android.defaultConfig.versionName`. Do not handwrite a different release version number.
- The versioned debug APK must be copied to `app/build/outputs/apk/debug/`, in the same directory as the default `app-debug.apk`. Do not create nested output folders such as `semantic/debug/`, `pixeldone/`, or other project-name/semantic directories.
- After `assembleDebug`, the same directory should contain both `app-debug.apk` and `{ProjectName}-{versionName}-debug.apk`.
- Deploy the versioned debug APK, not `app-debug.apk`, unless the user explicitly overrides the rule.

When a debug build must coexist with the installed formal app:

- Add `applicationIdSuffix = ".debug"` to the debug build type.
- Give the debug app the visible launcher label `PixelDone-beta`.
- Treat debug and formal builds as separate apps. They have separate local data, notification channels, alarms, permissions, and update behavior.

### 13.6 Formal Signed Release APK Rule

Private signed release APKs are the default channel for direct app-internal updates through GitHub Releases, with the synced Gitee release mirror as fallback.

Release APK rules:

- The app must use a long-lived release signing key, not the Android debug key.
- Release signing keystores and passwords must not be committed to Git.
- Release APK file name is always `{ProjectName}-{versionName}-release.apk`, for example `PixelDone-1.2.0-release.apk` or `PixelRead-1.0.0-release.apk`.
- `{versionName}` must come directly from the current app module's `android.defaultConfig.versionName`.
- The versioned release APK must be copied to `app/build/outputs/apk/release/`, in the same directory as the default `app-release.apk`.
- After `assembleRelease`, the same directory should contain both `app-release.apk` and `{ProjectName}-{versionName}-release.apk`.
- The formal app-internal updater should target normal GitHub Releases first and locate the versioned signed release APK asset. If GitHub checking or downloading fails, stalls, or lacks the matching APK asset, it may fall back to the synced Gitee mirror. It should not use local debug APKs as formal update assets.
- Direct APK distribution notes must tell users to uninstall incompatible debug/prototype builds when the package name or signing certificate changes.

### 13.7 Pre-Wrap-Up Checks

Pre-wrap-up checks:

- Run `git status -sb` from the PixelDone repository and confirm it only contains intended changes for the current release.
- Run the core tests.
- For local debug validation, run `assembleDebug` and confirm both debug APK files exist in `app/build/outputs/apk/debug/`.
- For formal release, run `assembleRelease`, confirm both release APK files exist in `app/build/outputs/apk/release/`, and verify signing with Android SDK signing tools when available.
- Confirm `README.md` feature descriptions, tech stack, build commands, install/update method, APK path, and current status match the real project and channel.

### 13.8 Commit Rules

Commit rules:

- Initial MVP commit message: `Initial {ProjectName} MVP`.
- Later pre-release or release-preparation commit messages should use a short verb phrase, for example `Refine EPUB reader layout`.
- Stage only files related to the current PixelDone change. Do not mix parent workspace files, build outputs, signing secrets, local IDE files, or generated cache files into a PixelDone commit.
- Before committing a version-number change, confirm the immediately previous numbered app version is already traceable to a commit whenever that version was built, installed, distributed, tagged, documented, or otherwise referenced.

### 13.9 GitHub Release Publishing And Gitee Update Mirror Notes

Formal signed release asset:

```text
app/build/outputs/apk/release/PixelDone-{versionName}-release.apk
```

Beta RC prerelease asset:

```text
app/build/outputs/apk/debug/PixelDone-{versionName}-debug.apk
```

Create the matching GitHub Release or prerelease at `https://github.com/Siyuan-Xue/PixelDone/releases`. After creating any GitHub Release, verify that the expected APK asset exists on GitHub.

The app update checker reads GitHub Releases first. It uses the synced Gitee mirror at `https://gitee.com/milesxue/PixelDone/releases` only as fallback for release checks and APK downloads. After the user's configured Gitee sync runs, verify that the expected APK asset also exists on Gitee so fallback update availability is healthy.
