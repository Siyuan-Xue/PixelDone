# PixelDone 2.4.4 Bug 修复教学案例

本文总结 PixelDone 2.4.4 中“代办条目跳转与高亮反馈”功能的 bug 修复历程。它面向不直接阅读代码的读者，重点解释问题现象、问题成因、修复思路和可复用经验。

## 1. 功能背景

这次迭代希望让用户在操作代办后更容易找到刚刚变化的条目。

目标体验可以概括为：

- 新增任务、编辑任务、取消完成任务、添加或替换图片后，列表跳到相关任务，并给任务边框高亮 2 秒。
- 完成任务时不跳转、不高亮，只保留原本的 2 秒延迟排序，方便用户连续勾选多个任务。
- 多个任务在同一个 2 秒窗口内连续变化时，统一处理；如果需要高亮，多条任务可以同时高亮。

为了方便理解，可以把这个功能拆成几个“角色”：

- 滚动指令：告诉列表“这次应该保持不动、回到顶部，还是显示某几个目标任务”。
- 高亮请求：告诉列表“这些任务需要亮一下边框，2 秒后自动消失”。
- 防抖批次：像一个临时篮子，2 秒内连续完成或取消完成的任务都先放进来，时间到后一次性处理。
- 视口保持：记录当前用户正在看的位置，排序变化后尽量让屏幕别突然跟着某个任务跑走。

## 2. Bug 一：取消完成失去防抖

现象：

用户连续把多个已完成任务改回未完成时，列表会立刻跳转和高亮，而不是等待 2 秒后统一处理。

成因：

完成任务和取消完成任务原本共用一套“勾选状态变化”的入口，但新增跳转逻辑后，取消完成被单独接到“立即跳转并高亮”的路径上。这样它绕过了原本为连续勾选设计的 2 秒防抖窗口。

修复思路：

把完成和取消完成重新放回同一个防抖批次里。

防抖批次里分两类记录：

- completedIds：本轮最终变成“已完成”的任务。
- undoneIds：本轮最终变成“未完成”的任务。

2 秒到期后再统一判断：

- 如果有取消完成任务，跳到其中位置最高的任务，并高亮这些任务。
- 如果只有完成任务，不跳转。

教学要点：

防抖不是简单的“延迟一下”，而是要保证所有相关动作都走同一个延迟入口。只要某个分支绕开了这个入口，连续操作体验就会碎掉。

## 3. Bug 二：完成任务有概率触发跳转

现象：

完成任务理论上不应该跳转，但有时完成任务后列表还是会跳到别的位置。

第一层成因：

列表滚动逻辑不只在“收到新的滚动指令”时运行，还会在可见条目顺序变化、排序冻结状态变化时重新运行。结果是：上一次新增、编辑或取消完成留下的旧滚动意图，可能在完成任务排序结束时被重新播放。

第一层修复：

让滚动处理只响应新的滚动序号。也就是说，只有真的发出一张新的“滚动票据”，列表才滚动。普通排序变化不会复用旧票据。

第二层边界：

完成列表第一条任务时，仍然可能跳转。

这不是旧滚动指令导致的，而是列表组件自身的行为：当列表按 key 识别每一行时，它会尽量保持“当前第一条可见任务”继续可见。如果第一条完成后被排序到后面，列表就可能跟着这条任务一起移动，看起来像发生了跳转。

第二层修复：

完成-only 的防抖结束时，不发“显示某个任务”的指令，而发“保持当前视口”的指令。列表会记住当前屏幕顶部大概是第几行、偏移多少，然后请求下一次布局继续停在这个位置。

教学要点：

“没有主动滚动”不等于“屏幕不会动”。现代列表组件会为了稳定 key 做自己的位置保持。遇到排序后的意外跳动，要同时检查业务滚动逻辑和列表组件默认行为。

## 4. Bug 三：批量取消完成后部分高亮永久不消失

现象：

批量取消完成多个任务时，最后一个高亮会正常消失，但其他任务的高亮可能留在屏幕上。

成因：

一开始高亮消失的计时器放在每一行自己内部。列表行会因为滚动、排序、重组而被销毁或重建。如果某一行的计时过程被中断，或者它不再属于最新高亮请求，它可能没有机会把自己的高亮状态关掉。

修复思路：

把“哪些任务正在高亮”交给列表父层统一管理：

- 收到高亮请求时，父层记录一组正在高亮的任务 ID。
- 2 秒后父层一次性清空这组 ID。
- 每一行只负责看“我的 ID 是否在这组 ID 里”，不再自己管理计时生命周期。

教学要点：

当多个列表项共享同一个反馈周期时，把生命周期放在父层通常更稳。列表项适合展示状态，不适合独立维护全局反馈计时。

## 5. Bug 四：高亮边框不够明显

现象：

高亮边框只有 1dp，在 PixelDone 的浅色背景和密集列表里不够醒目。

成因：

视觉反馈强度不足。虽然颜色是强调色，但线宽太细，用户快速扫视时不容易注意到。

修复思路：

高亮期间边框从 1dp 提升到 2dp，颜色继续使用原有强调色。这样不改变整体风格，也不会让列表行高度明显跳动。

教学要点：

反馈动画不是越多越好，但关键反馈必须能被看见。对于工具类应用，清晰的边框变化往往比复杂动画更可靠。

## 6. Bug 五：附图预览里的关闭按钮多余

现象：

图片预览弹窗底部有关闭按钮，但用户点击弹窗外区域已经可以关闭。

成因：

界面里存在两个功能重复的退出路径。关闭按钮占用了操作区注意力，也让 CHANGE 和 REMOVE 两个真正的图片操作显得不够突出。

修复思路：

移除预览弹窗底部的关闭按钮，保留点击弹窗外关闭。CHANGE 和 REMOVE 仍然保留。

教学要点：

如果一个操作可以通过系统习惯自然完成，界面里就不一定需要再放一个按钮。少一个重复控件，用户反而更容易看懂真正重要的动作。

## 7. Bug 六：完成任务不应该高亮

现象：

完成任务虽然已经不跳转，但仍然可能出现边框高亮。

成因：

防抖批次里同时记录了完成任务和取消完成任务，而高亮目标曾经直接取“两者并集”。这让完成任务也被当成需要强调的目标。

修复思路：

高亮目标只取取消完成任务。完成任务仍然记录在批次中，用来触发延迟排序和保持视口，但不进入高亮列表。

教学要点：

同一个批次里可以有多种用途的数据。不要因为它们都在一个篮子里，就默认它们都应该得到同一种反馈。

## 8. Bug 七：最后一个高亮条目离屏后再出现会重新高亮

现象：

某个条目高亮结束后，如果它曾经离开屏幕再回来，可能又重新触发一次高亮。

成因：

当高亮序号保存在行上时，列表项重建后会看到“这个序号不是 0”，于是又从头播放高亮动画。对用户来说，旧反馈被误认为新反馈。

修复思路：

高亮是否有效不再由单个行根据旧序号决定，而是由父层维护一个 2 秒有效期的“当前高亮集合”。集合被清空后，离屏再回来的行也只能看到“我不在高亮集合里”，因此不会重新亮起。

教学要点：

列表项会被复用、销毁、重建。只要反馈有时间窗口，就要警惕“组件重建导致旧动画重放”。

## 9. 最终行为总览

2.4.4 最终形成的规则是：

- 新增任务：跳到目标任务并高亮。
- 编辑任务：跳到目标任务并高亮。
- 添加或替换图片：回到列表、跳到目标任务并高亮，不自动弹出图片预览。
- 删除图片：保持当前位置，不跳转。
- 取消完成：进入 2 秒防抖窗口，结束后跳到本批取消完成任务里位置最高的任务，并高亮本批取消完成任务。
- 完成任务：进入 2 秒防抖窗口，结束后只恢复排序，不跳转、不高亮，并尽量保持当前屏幕位置。
- 同一个任务在防抖窗口内反复切换：以最后一次状态为准。

## 10. 可复用经验

这次修复可以提炼成五条经验：

1. 把“用户动作”和“反馈策略”分开看。同样是勾选变化，完成和取消完成需要不同反馈。
2. 防抖批次要记录最终状态，而不是记录每一次中间动作。
3. 滚动逻辑必须有明确的一次性触发信号，避免旧滚动请求在排序或重组时重放。
4. 多个列表项共享的反馈生命周期应放在父层管理。
5. 列表排序后的跳动不一定来自业务代码，也可能来自列表组件为了保持 key 稳定而做的默认行为。
